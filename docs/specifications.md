# RPG Dungeon Crawler
This project is a top-down 2D Dungeon Crawler where you go through multiple randomised floors killing enemies and looting weapons and equipment.

We'll also try to add RPG elements like character and class selection, friendly/neutral characters and quests.

Repo used: https://github.com/IndaPlus25/msorberg-korlot-project

Project manager used: https://github.com/orgs/IndaPlus25/projects/10

# Goals
Our MVP should be focused on having:
- A controllable player
- A weapon to attack enemies
- Enemies (of a single type)
- Randomly generated rooms

If we have more time then we should focus on implementing:
- Multiple weapons
- Multiple enemy types
- Multiple floors
- Looting
- Rogue-like system
  - Cash system
  - Upgrades
- Audio system
- Pixel art graphics
- RPG system
  - Character selection
  - Class selection
  - Friendly/neutral characters
    - Attacks you if you attack them
    - If you attack way to many they automatically attack you
  - Shopkeepers
  - Quests

# Collaboration
This will cover how we will divide up the project, what project manager we will be using it, how we will be using the project mangaer and what specific naming conventions will be used for issues and branches.

## Github projects
We will be using github projects for this project. Making use of its kanban board and roadmap.

### <ins>Columns:

We will divide up the board into 7 different columns: 
- Ideas
- Optional Implementations
- TODO
- In progress (Korlot)
- In progress (Msorberg)
- In review
- Complete

`Ideas` will have all our ideas that we haven't agreed upon yet or aren't sure would fit our game. 

If we both agree it's a good idea then it either moves into `TODO` or `Optional Implementations` depending on if we know we have to implement it or not.

`In progess (Korlot)` & `In progress (Msorberg)` is to keep track of what the other person is doing. When we have started working on an issue we pull that issue from `TODO` into our respective column to show the other person that we started working on it. 

Once we are done we can move it into `In review`. Every issue needs to be reviewed by both of us before it can be pushed into `main`. 

After it's been done and the issue can be pushed to main it will finally be moved to `Complete`.

### <ins>Roadmap:
Every friday we will have a milestone with issues to be fixed until then. Every issue will be marked with that milestone to be easily shown in the roadmap and board.

## Issues
We will be using future tense and capital letter in the beginning, like `Fix this bug`, to follow Github Standard.

One issue will cover a pretty big category of issues like `Add a player`, `Add weapons` or `Add random room generation`.

Subissues like `Add jump` or `Create rooms` will be put inside the description of that issues with a checkbox. 

The description will also have a quick description to explain which type of issues it will solve.
### <ins>Example:
### Add a player #1
Create a player
- [ ] Add jump

## Branches
One branch will be created for each issue. 

The branch name will describe the feature in one or two words in kebab-case, like `player`, `enemy`, `weapons` or `random-room-generation`. 

Once one of the features are completed they will be pushed to main using pull requests.
